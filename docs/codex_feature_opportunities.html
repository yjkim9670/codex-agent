3파일럿 후 판단

현재 워크벤치 기준선

워크벤치는 Python/Flask 서버가 브라우저 UI, 파일 브라우저, 터미널, Git 동기화, 사용량 모니터링을 제공하고, Codex 실행은 codex exec --json 프로세스를 띄워 stdout JSONL과 stderr를 스트리밍하는 방식입니다. 현재 일반 실행은 codex --ask-for-approval never exec --sandbox workspace-write, question-only 실행은 read-only --ephemeral 경로를 사용합니다.

강점

  • 이미 JSONL 이벤트, 토큰 사용량, 원격 스트림 재연결, queue wait, runtime 상태를 수집합니다.
  • 브라우저 UI와 Git overlay, usage overlay가 있어 기능 추가 지점이 명확합니다.
  • --full-auto 없이 명시적 sandbox를 사용하고, queued Codex home을 준비하는 안전 장치가 있습니다.

제약

  • 현재 thread store와 Codex 내부 thread lifecycle을 직접 읽는 구조는 아닙니다.
  • subagent, model catalog, experimental feature flag, rollback/fork는 JSONL 이벤트만으로 표현하기 어렵습니다.
  • 명시적 sandbox 전환은 완료됐지만, UI에서 approval/sandbox 프리셋과 위험도를 선택하거나 비교하는 화면은 아직 없습니다.

기능 문서상 핵심 워크벤치 적용 의미
Codex CLI 0.130.0 2026-05-08 릴리스에서 codex remote-control, app-server thread pagination, plugin hook 표시, app-server config refresh 등이 추가되었습니다. 이미 설치된 0.130.0을 기준으로 remote-control 파일럿을 시작할 수 있습니다.
Codex SDK TypeScript SDK는 local Codex agent를 서버 측에서 제어하고, thread start/run/resume을 제공합니다. Python SDK는 experimental이며 local app-server JSON-RPC를 제어합니다. Python 서버에 바로 이식하기보다는 TypeScript sidecar 또는 app-server JSON-RPC 어댑터가 현실적입니다.
App Server JSON-RPC로 thread start/resume/fork/read/list, compact, rollback, inject items, model/list, experimentalFeature/list를 제공합니다. 워크벤치가 직접 thread lifecycle UI를 만들 수 있는 가장 큰 후보입니다.
Subagents 현재 Codex 릴리스에서 기본 활성화되어 있으며, 사용자가 명시적으로 요청할 때만 spawn합니다. built-in agent는 default, worker, explorer입니다. UI에서 “병렬 조사/구현”을 명시 버튼으로 제공하고 비용 경고를 같이 보여주는 형태가 적합합니다.
Skills와 Plugins Skills는 SKILL.md, scripts, references, assets로 반복 워크플로를 패키징합니다. Plugins는 skills, apps, MCP servers를 묶는 설치 단위입니다. 워크벤치에 repo skill authoring, 설치 상태 보기, task preset 생성을 추가할 수 있습니다.
Hooks, Rules, MCP Hooks는 lifecycle event에 command를 연결하고, Rules는 sandbox 밖 명령을 prefix 단위로 allow/prompt/forbid합니다. MCP는 third-party tools와 context를 연결합니다. 워크벤치 안전 정책과 외부 도구 연결을 UI에서 관리하는 설정 패널 후보입니다.
Worktrees와 Automations Codex app은 독립 작업을 Git worktree에서 병렬 수행하고, automations는 background worktree에서 반복 작업을 수행할 수 있습니다. 워크벤치에서도 Git worktree 기반 격리 실행과 예약 task를 자체 구현할 가치가 큽니다.
Non-interactive structured output codex exec --json은 JSONL 이벤트를 내보내고, --output-schema는 최종 응답을 JSON Schema에 맞게 받을 수 있습니다. 이미 쓰는 JSONL 구조를 확장해 PR 리스크, 테스트 계획, 릴리즈 노트 같은 안정적인 결과물을 만들 수 있습니다.
Codex Models Codex 문서는 gpt-5.5를 복잡한 coding, computer use, knowledge work, research workflow에 추천하지만 ChatGPT 로그인 환경에서 제공되며 API-key auth에서는 제공되지 않는다고 설명합니다. 모델명을 하드코딩하지 말고 app-server model/list나 CLI 설정을 통해 계정별 가용성을 보여주는 UI가 필요합니다.

업데이트 리스크 헤지

바로 App Server나 SDK 중심 구조로 갈아타지 않고, 현재 안정 경로를 유지한 채 새 기능을 작게 붙이는 방식이 안전합니다.

리스크 헤지 방법 완료 기준
실행 경로 회귀 codex exec --json을 기본 경로로 유지하고, App Server/remote-control은 feature flag 뒤의 별도 어댑터로 둡니다. 실패 시 기존 exec 경로로 즉시 fallback합니다. 기존 스트림, usage, queue, reconnect 테스트가 그대로 통과하고 flag off 상태에서 동작 차이가 없습니다.
권한 확대 현재 workspace-write와 question-only read-only --ephemeral 조합을 기준선으로 고정하고, sandbox 밖 명령이나 thread/shellCommand 류 기능은 초기에 UI에 노출하지 않습니다. 실행 커맨드 스냅샷 테스트가 --full-auto 미사용과 sandbox 값을 검증합니다.
작업물 충돌 새로운 구현/수정 작업은 Git worktree에서 실행하고, 본 checkout에는 diff review, handoff, cleanup 버튼만 제공합니다. dirty worktree가 있는 상태에서도 독립 작업 생성, diff 확인, 폐기가 가능합니다.
출력 파싱 불안정 structured report는 --output-schema 최종 응답만 엄격 검증하고, 중간 JSONL 이벤트 파서는 현재 tolerant parser를 유지합니다. schema validation 실패가 전체 스트림 실패로 번지지 않고 사용자에게 재시도 가능한 오류로 표시됩니다.
운영 비용 증가 subagent 병렬 실행은 자동 fan-out이 아니라 명시 버튼과 최대 병렬 수, 예상 token 표시를 둔 프리셋으로 시작합니다. 사용자가 병렬 수와 모델을 확인한 뒤에만 subagent 실행이 시작됩니다.

구현 시 주의점

App Server 전환 리스크
App Server는 워크벤치가 가장 원하는 thread lifecycle 기능을 제공하지만, 현재 구조를 즉시 대체하면 stream 처리, 세션 저장, 인증, 프로세스 생명주기, queue 정책이 동시에 바뀝니다. 읽기 전용 API부터 붙이는 것이 안전합니다.
Subagent 비용 리스크
공식 문서는 subagent가 각자 모델과 도구 작업을 수행하므로 단일 agent보다 token을 더 쓴다고 설명합니다. UI에는 예상 비용, 최대 병렬 수, 명시 실행 조건이 필요합니다.
  • 모델 기본값은 gpt-5.5 같은 이름으로 하드코딩하지 말고 계정별 가용성을 조회하거나 현재 설정을 존중해야 합니다.
  • Hooks, Rules, MCP config는 사용자의 Codex 전역 환경을 바꿀 수 있으므로 프로젝트 스코프 preview와 명시 적용 버튼으로 시작하는 것이 좋습니다.
  • Git worktree는 branch checkout 제약과 ignored file handoff 제약이 있으므로 UI에서 “무엇이 이동하지 않는지”를 보여줘야 합니다.
  • GitHub Action 템플릿은 secret, permissions, safety-strategy를 포함하므로 자동 커밋보다 preview 생성이 안전합니다.

결론

워크벤치의 다음 기능 추가는 App Server 전체 전환보다 “기존 exec 기반 안정화 + worktree 격리 + structured reports”를 먼저 잡는 것이 가장 실용적입니다. 그 다음 remote-control/App Server를 별도 어댑터로 붙여 thread lifecycle과 model catalog를 검증하면, 현재 구현을 크게 흔들지 않으면서 최신 Codex 기능의 이점을 단계적으로 가져올 수 있습니다.
